# 学习周报 | Git分布式版本控制工具

## 一、 本周学习目标回顾
本周的学习核心为全面掌握**Git分布式版本控制工具**。预期目标包括：
1. 深入理解Git的基本概念及分布式与集中式（SVN）版本控制的核心差异。
2. 熟练掌握Git工作流程（工作区、暂存区、本地仓库、远程仓库）的数据流向。
3. 熟练运用Git常用命令行（本地仓库配置、基础提交、版本回退、分支管理及冲突解决）。
4. 掌握代码远程托管服务（Gitee），实现本地与远程的多人协同开发。
5. 掌握如何在IDEA中深度集成并可视化操作Git，对齐企业实际开发规范。

## 二、 每日学习进度与详实内容

### 星期一：Git概述、安装与个性化环境配置
*   **版本控制对比认知**：学习了集中式（SVN/CVS）与分布式（Git）的本质区别。集中式严重依赖中央服务器，无网无法工作且存在单点崩溃风险；而分布式每个人的电脑都是一个完整的版本库，本地即可完成版本控制，多人协同只需推送各自修改。
*   **环境搭建与初始化**：下载并安装了Git Windows版本，了解了Git GUI（图形化工具）与Git Bash（命令行工具）的区别。
*   **全局用户信息配置**：由于Git版本控制需要辨识提交者，安装后第一时间通过全局命令配置了开发者的用户名和邮箱。
*   **效率黑科技-指令别名（Alias）**：为了避免频繁输入冗长的日志查看参数，在用户目录下通过 `touch ~/.bashrc` 创建了配置文件，并配置了高效的图形化精简日志别名 `git-log`。
*   **乱码问题一键修正**：针对Windows环境下Git Bash可能出现的中文路径与终端乱码，通过配置 `core.quotepath` 以及修改 `${git_home}/etc/bash.bashrc` 环境变量，彻底解决中文显示异常问题。

### 星期二：Git本地仓库创建与核心基础指令
*   **本地仓库初始化**：在本地空目录下通过 `git init` 成功将目录初始化为Git本地仓库，生成了隐藏的 `.git` 目录（版本库核心）。
*   **文件状态追踪与转换**：深入学习了工作目录下文件的四种状态：未跟踪（untracked）、已暂存（staged）、已修改（modified）、未修改。
*   **基础操作闭环演练**：
    *   使用 `git status` 实时查看工作区和暂存区的修改状态。
    *   使用 `git add` 将工作区的新建或修改文件提交至暂存区（index）。
    *   使用 `git commit -m "注释内容"` 将暂存区的缓存提交至本地仓库，形成永久的历史版本记录。
    *   使用 `git log` 或自定义的 `git-log` 别名查看完美的版本演进线索。
*   **时光机-版本回退与还原**：掌握了利用 `git reset --hard commitID` 实现版本任意切换。若误删版本后，学到了利用 `git reflog` 逆向查找已被删除的提交记录ID，完成版本“起死回生”的还原操作。
*   **文件忽略机制（.gitignore）**：学习了如何编写 `.gitignore` 规则文件，用于强制排除自动生成的日志文件、编译临时文件（如 `.a`、`build/`）等，使其不进入未跟踪列表，保持工作区干净。

### 星期三：Git分支管理、多分支协同与冲突彻底解决
*   **分支的概念与威力**：分支能够将个人的开发工作从主线上分离开来，在不影响主线（`master`）稳定性的前提下开发新功能或修复Bug。
*   **分支基础命令**：掌握了查看本地分支（`git branch`，带 `*` 号代表当前所在分支）、创建分支（`git branch 分支名`）、切换分支（`git checkout 分支名`）以及一步到位的创建并切换命令（`git checkout -b 分支名`）。
*   **主线合并（Merge）**：完成分支业务开发并提交后，需先切换回 `master` 主线，再通过 `git merge 分支名` 将其合并。合并完成后，可通过 `git branch -d` 安全删除该临时分支（若分支未合并则需用 `-D` 强制删除）。
*   **核心难点：手动冲突解决**：当两个分支对同一个文件的**同一行**进行不同修改并尝试合并时，Git会报 `CONFLICT (content)` 错误，文件内会自动注入冲突标记（`<<<<<<< HEAD` 当前分支修改、`=======` 分割线、`>>>>>>> dev` 冲突分支修改）。学习了必须通过手动编辑文件，保留最终期望代码，删掉标记符号，然后重新执行 `git add` 和 `git commit` 方能闭环冲突。

### 星期四：码云Gitee远程仓库对接与多端数据同步
*   **云端代码托管**：了解了主流托管平台（GitHub、Gitee、GitLab），并在码云Gitee上完成了账户注册与全新的远程仓库（Remote Repository）创建。
*   **免密安全通道-SSH公钥配置**：利用 `ssh-keygen -t rsa` 在本地连续回车生成密钥对，通过 `cat ~/.ssh/id_rsa.pub` 读取公钥并粘贴到Gitee后台。最终通过 `ssh -T git@gitee.com` 成功验证安全通信通道。
*   **远程仓库 CLI 交互**：
    *   通过 `git remote add origin <SSH路径>` 实现本地仓库与云端远程仓库的高效绑定（默认远端名称为 `origin`）。
    *   使用 `git push --set-upstream origin master:master` 将本地主线推送到云端，并建立两端分支的长期关联。后续可直接精简为 `git push`。
    *   利用 `git branch -vv` 随时自检本地分支与远程分支的绑定映射关系。
*   **多机协作与拉取同步**：新入职或多机开发时，使用 `git clone <仓库路径>` 完整克隆云端版本库。重点区分了 `git fetch`（仅抓取云端更新，不自动合并，本地看不到新内容）与 `git pull`（抓取并自动合并，等同于 `fetch + merge`）的区别。明白协同开发时，必须确保本地版本最新（先 `pull`）才能允许 `push`。

### 星期五：IntelliJ IDEA中深度集成Git与企业级分支模型实战
*   **IDEA 全局配置**：在IDEA Settings -> Version Control -> Git 中指定 `git.exe` 路径，点击 "Test" 出现版本号即表示集成成功。
*   **文件颜色状态完美对齐**：
    *   `绿色`：已add进入暂存区，等待提交。
    *   `爆红`：未跟踪的新建文件，或处于合并冲突状态。
    *   `灰色`：已被 `.gitignore` 明确忽略的文件。
*   **图形化闭环操作**：掌握了在IDEA中通过右上角快捷按钮或底端 "Git" 选项卡实现：可视化 Commit、Push、Clone、创建/切换分支（右下角）。同时演练了IDEA强大的三栏式可视化冲突解决工具（Accept Yours / Accept Theirs / Merge）。
*   **企业开发分支模型原则**：
    *   `master`（生产分支）：严禁直接修改，专门对应线上平稳运行的多版本节点。
    *   `develop`（开发分支）：从master派生，所有组员的主要合并基线，大版本上线时才合并回master。
    *   `feature/xxxx`（特性分支）：为某个并行独立功能创建，完工后合并到develop，用完可删。
    *   `hotfix/xxxx`（紧急修复分支）：线上主线出bug时紧急从master派生，修复测试完必须同时合并到master、develop及test分支。

## 三、 本周学习问题与解决方案
1.  **Git Bash 中文乱码**：Windows下部分中文路径显示为八进制编码。
    *   *解决办法*：执行了全局配置 `git config --global core.quotepath false`，并在Git安装目录下的 `etc/bash.bashrc` 中补全了 `export LANG="zh_CN.UTF-8"` 及 `export LC_ALL="zh_CN.UTF-8"`，重启终端后显示完美。
2.  **分支删除失败**：在master分支尝试删除一个刚写完代码的分支时报错。
    *   *原因剖析*：该分支的最新修改尚未合并到当前分支，Git为了防止代码丢失会触发安全保护。
    *   *解决办法*：确认代码确实不需要或已手动备份后，将 `-d` 换成了大写的 `-D` 强制删除指令：`git branch -D dev01`。
3.  **远程推送被拒绝（Rejected）**：在本地写完代码直接 `git push` 结果报错拒绝。
    *   *原因剖析*：团队中其他成员已经优先向远程仓库提交并推送了相同文件的修改，导致远程版本领先于本地版本。
    *   *解决办法*：遵循“先拉后推”铁令，先执行 `git pull` 把远程最新版拉下来，若有冲突则在本地手动（或利用IDEA可视化）把冲突解决并重新commit，之后再安全执行 `git push`。
